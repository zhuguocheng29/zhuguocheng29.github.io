1. 修复全面屏下超出uiautomator识别范围区域的tap和double tap操作(支持元素,图像,坐标)

排查问题

问题出现条件

在可以切换到全面屏模式下的手机切换到全面屏模式后，在某些区域（如NavigationBar底部），无法进行点击操作

目前，脚本执行时，整个流程如下（基本流程图如下）： 基于mocha的脚本（服务端PC） -> macaca-wd -> macaca server -> uiautomator-client -> SDK(手机端) -> 手机

解决问题过程（目前）

  1. 通过查看macaca server的log后发现，相关的点击信息已经通过json的形式传递给了macaca server。然而macaca server迟迟没有将uiautomator-client的信息传回来，最后造成脚本卡主，直至10分钟后退出。所以问题很快就锁定在uiautomator-client中。
  2. 用Android Studio打开Uiautomatorclient目录查询原因。最后查到原来在uiautomator-client中，当设备的真正的deviceHeight != 设备去掉NavigationBar的deviceHeight时，直接返回false，不做任何的操作。
  3. 原来如此。那就可以在上述的情况下增加对应的操作。难点来了！UIAutomator中的UIObject2这个类是不可修改的，那怎么办呢？反射。天呐准备第一次在项目中实现反射了，原先一直只是听说但并未实战过。此刻非常感谢阿里团队，他们将一些方法对应的反射进行了封装，每次只需要选择对应的方法进行调用即可，大大减少了开发的难度。此刻在全面屏下基于图像和坐标的点击大功告成。
  4. 可是我们业务的大部分场景还是基于元素进行点击，上述已经实现的的方法能够支持基于元素的点击吗？答案是：不支持。怎么支持呢？将元素对象转换成相对应的坐标！此时真正的坑点来了，你会发现你没法获得元素的信息. what? 再看源码，发现是缺少一个refresh()操作。此时，也验证了之前的一个问题，即在全面屏下，有时候我们点不上对应的位置，但是点击下屏幕之后就可以点到对应的位置了，之前一直很有疑惑，现在终于知道了。但是为什么全面屏在必须先点refresh()操作呢，目前不太知道，得后面看看源码了

通过driver.initDriver(desiredCapabilities)这个操作启动driver,这个过程便会想macaca server发送http请求，server会接收到这个请求并创建一个session，在这次的用例执行中，所有的操作都会基于这个session进行

为什么不用stf的方式呢？ 个人觉得一方面是不想让回放和执行是两套代码逻辑，增加开发和维护的复杂度。后面问了一下mentor,mentor的意思是stf的方式难用于单元测试，而macaca可以用于单元测试，再配合mocha是一个较好的方案。依旧表示疑问？持续追踪

2. XPath 优化

什么是XPath?

XPath是XML路径语言，是一种用来准确定位XML文档中某部分位置的语言。使用XPath可以来选取XML文档中的节点和节点集。

因为Uiautomator是基于XML的，所以我们可以通过XPath找到对应的节点。

目前，我们所有XPath路径的生成都基于原始的基本XPath路径生成方法

基本思想:自该节点起向上遍历,直到找到能唯一标识该节点的XPath路径为止,其中index都从1开始.

多种XPath路径排序

XPath优先级从上往下

  1. 没有index的在前,有index的在后.
  2. 在没有index的结果中:有text或content-desc内容且该内容小于50字符最靠前(需排序).
  3. 在没有index的结果中,对于超出长度的情况:text的长度跟其余没有index长度的进行比较,越小越靠前
  4. 在有index的结果中,有text或context-desc内容的靠前,其余的靠后
